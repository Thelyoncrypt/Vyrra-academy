## 1. Hermes Agent: Detailed Architecture

### 1.1 The Closed Learning Loop

Hermes Agent's defining architectural feature is its **closed learning loop** — a self-improvement cycle where the agent automatically reflects on completed tasks, extracts reusable patterns, and writes them as skill files. This is what makes it "the agent that grows with you." [^18^] [^22^] [^66^]

**How the learning loop works:**

1. **Task Execution**: The agent completes a complex task (typically 5+ tool calls)
2. **Reflection Trigger**: After completion, a reflection module evaluates whether the task pattern is reusable
3. **Skill Distillation**: Execution experience is distilled into a Markdown skill file following the agentskills.io open standard
4. **Progressive Disclosure**: Skills are stored at three levels:
   - **Level 0**: ~3000 token summary (quick retrieval)
   - **Level 1**: Full skill content
   - **Level 2**: Reference materials and linked resources
5. **Reflection Cycle**: Every 15 tasks, a reflection loop evaluates existing skills for effectiveness and improves them [^22^] [^29^] [^66^]

This contrasts sharply with OpenClaw, where skills are static Markdown files handwritten by users and distributed through the ClawHub marketplace. Hermes skills are auto-generated by the agent itself, eliminating the third-party supply-chain attack vector entirely. [^34^] [^66^]

### 1.2 Major Subsystems

The Hermes Agent architecture consists of 10+ major subsystems [^321^]:

| Subsystem | Purpose |
|---|---|
| **Agent Loop** | Synchronous orchestration engine handling provider selection, prompt construction, tool execution, retries, fallback, callbacks, compression, and persistence |
| **Prompt System** | Assembles system prompt from personality (SOUL.md), memory (MEMORY.md, USER.md), skills, context files, tool-use guidance, and model-specific instructions |
| **Provider Resolution** | Shared runtime resolver mapping (provider, model) to (api_mode, api_key, base_url) — handles 18+ providers, OAuth flows, credential pools |
| **Tool System** | Central registry with 70+ registered tools across ~28 toolsets; terminal tools support 7 backends |
| **Session Persistence** | SQLite-based storage with FTS5 full-text search; lineage tracking, per-platform isolation, atomic writes |
| **Messaging Gateway** | Long-running process with 20 platform adapters, unified session routing, user authorization, slash commands |
| **Plugin System** | Three discovery sources; plugins register tools, hooks, and CLI commands |
| **Cron Scheduler** | First-class agent tasks with JSON storage, multiple schedule formats, skill attachment |
| **ACP Integration** | Exposes Hermes as editor-native agent over stdio/JSON-RPC for VS Code, Zed, and JetBrains |
| **Trajectories** | Generates ShareGPT-format trajectories from sessions for training data generation |

### 1.3 Subagent Delegation Architecture

Hermes supports native subagent delegation for parallel workstreams [^372^] [^373^] [^375^]:

**Current Capabilities:**
- **Default concurrency**: 3 parallel subagents (configurable, no hard ceiling)
- **Max delegation depth**: 3 levels (parent -> orchestrator -> leaf)
- **Isolation**: Each subagent gets its own conversation, terminal session, and toolset
- **Separation**: Parent and subagents share no conversation history; subagents see only the `goal` and `context` passed at creation

**Inter-Agent Communication Levels** (roadmap) [^372^]:

| Level | Mechanism | Use Case |
|---|---|---|
| **L0: Isolated** (current) | No sharing, parent relays | Simple delegation |
| **L1: Result passing** | Upstream results auto-injected into downstream context | Workflow DAGs |
| **L2: Shared scratchpad** | Read/write shared key-value store | Complex workflows |
| **L3: Live dialogue** | Turn-based agent-to-agent conversation | Debate/review modes |

**Configuration example:**
```yaml
delegation:
  max_iterations: 50                        # Max turns per child
  max_concurrent_children: 3               # Parallel children per batch
  max_spawn_depth: 1                       # Tree depth (1-3)
  orchestrator_enabled: true               # Enable nested delegation
  model: "google/gemini-3-flash-preview"   # Override model for subagents
  provider: "openrouter"
```

### 1.4 Skill Auto-Generation in Detail

When Hermes finishes a complex task successfully, it synthesizes the approach into a permanent Markdown file [^326^]:

```
~/.hermes/skills/
  ├── github-trending-analysis.md
  ├── server-health-check.md
  ├── research-pipeline.md
  └── ...
```

Each skill follows the agentskills.io open standard and includes:
- Task description and trigger conditions
- Step-by-step execution pattern
- Required tools and their configurations
- Expected inputs and outputs
- Lessons learned from previous executions

These files are human-readable, immediately reusable, and shareable through the community skills hub. They also self-improve during use — more repetitions of similar tasks produce more refined skill files. [^326^]

## 3. Setup Guide: Installation & First Workflows

### 3.1 Installation

**One-line install** (macOS, Linux, WSL2) [^307^] [^308^]:
```bash
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
source ~/.bashrc   # or: source ~/.zshrc
hermes             # start chatting
```

**Windows native** (early beta):
```powershell
irm https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.ps1 | iex
```

**What the installer sets up:**
- Python 3.11 and uv package manager
- Node.js dependencies
- Chromium for browser automation (via Playwright)
- ripgrep, ffmpeg, and other system tools
- `~/.hermes/` directory tree (config, memory, skills, logs, sessions)
- Symlinks `hermes` to `~/.local/bin/`

**Verify installation:**
```bash
hermes version      # v0.10.0 or newer
hermes doctor       # System health check — every line should be green
hermes status       # Current config + running daemons
```

**Quick uninstall/reinstall:**
```bash
rm -rf ~/.hermes/   # Complete removal including memory and skills
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
```

### 3.2 Manual Installation

For step-by-step control [^307^]:
```bash
git clone --recurse-submodules https://github.com/NousResearch/hermes-agent.git
cd hermes-agent
curl -LsSf https://astral.sh/uv/install.sh | sh
uv venv venv --python 3.11
export VIRTUAL_ENV="$(pwd)/venv"
uv pip install -e ".[all]"
npm install
mkdir -p ~/.hermes/{cron,sessions,logs,memories,skills,pairing,hooks,image_cache,audio_cache}
cp cli-config.yaml.example ~/.hermes/config.yaml
touch ~/.hermes/.env
ln -s "$(pwd)/bin/hermes" ~/.local/bin/hermes
```

### 4.2 DeepHermes 3: Dual-Mode Reasoning

DeepHermes 3 is one of the first models to unify both "intuitive" (traditional) response mode and "deep reasoning" (long chain-of-thought) mode into a single model, toggled by system prompt. [^374^] [^379^] [^381^]

**To toggle reasoning ON**, use this system prompt:
```
You are a deep thinking AI, you may use extremely long chains of thought to deeply 
consider the problem and deliberate with yourself via systematic reasoning processes 
to help come to a correct solution prior to answering. You should enclose your 
thoughts and internal monologue inside <think> </think> tags, and then provide 
your solution or response to the problem.
```

**Key characteristics:**
- Based on Mistral-Small-24B, fine-tuned via distillation from R1
- Supports structured output (JSON mode) and function call syntax
- Can use up to 13,000 tokens for deep thinking on difficult problems
- Uses Llama-Chat prompt format
- Available via Hugging Face, OpenRouter, and Nous API portal [^379^] [^381^]

**Running with vLLM:**
```bash
vllm serve NousResearch/DeepHermes-3-Mistral-24B-Preview
```

| Dimension | Hermes 3 | Hermes 4 |

## 6. Persistent Memory Architecture

### 6.1 The Three-Layer Memory System

Hermes Agent's memory architecture is its key differentiator. Unlike most agents that are stateless by default, Hermes ships with persistent cross-session memory using SQLite + FTS5. [^320^] [^321^] [^322^]

**Layer 1 — Session Context (In-Process):**
- Current conversation buffer, tool outputs, scratch space
- Stored in-process only
- Lifetime: current session

**Layer 2 — Persistent Store (SQLite + FTS5):**
- Completed task outcomes, generated skill files, user notes
- SQLite with WAL mode and FTS5 full-text index
- Stores every prior conversation turn
- ~10ms retrieval latency for 10,000+ documents
- Lifetime: forever (backed by `~/.hermes/`)

**Layer 3 — User Model (SQLite JSON field):**
- Preferences, coding style, timezone, tone, frequent collaborators
- Drift-adjusted across sessions
- Agent automatically decides what to remember and what to forget

### 6.2 The Two Markdown Memory Files

Two autonomous markdown files managed by the agent itself [^327^]:

| File | Purpose | Size Cap |
|---|---|---|
| `MEMORY.md` | Environment facts, lessons learned | ~2,200 characters |
| `USER.md` | User preferences, identity model | ~1,375 characters |

These files are:
- **Agent-curated**: The agent decides what deserves persistence without manual configuration
- **Frozen snapshots**: Read once at session start and embedded in system prompt as immutable block
- **Writable mid-session**: Agent can update them during conversation, but changes only appear in next session
- **Prefix-cache friendly**: The frozen snapshot design keeps the Anthropic cache breakpoints valid

### 6.3 FTS5 Full-Text Search

The SessionDB uses SQLite FTS5 for cross-session recall [^327^]:

```python
# On demand, the agent uses session_search tool to query
# LLM summarizer condenses hits into context-fitting paragraph

# Technical details:
# - SQLite WAL mode for multi-process write contention
# - BEGIN IMMEDIATE + custom retry loop (20-150ms jitter)
# - FTS5 virtual table on message text column
# - LLM summarization of search results before injection
```

**Why SQLite + FTS5 over vector databases?** Nous Research explicitly chose this for the embedded-first deployment story — the whole agent ships as files under `~/.hermes/`, with no external database dependencies. This keeps retrieval latency at ~10ms for 10,000+ entries, the inflection point where most vector DB architectures show tail latency. [^322^]

### 6.4 Honcho User Modeling

Layer 3 incorporates Honcho for user modeling — tracking preferences, coding style, timezone, tone, and frequent collaborators. This model drifts gradually across sessions, with the agent automatically evaluating what information deserves long-term persistence. [^322^]

## 7. Multi-Platform Gateway

### 7.1 Gateway Architecture

Hermes Agent's messaging gateway is a single long-running process that connects to multiple messaging platforms simultaneously, enabling cross-platform conversation continuity. [^323^] [^321^]

```
flowchart:
    [Hermes Gateway] --> [Telegram]
    [Hermes Gateway] --> [Discord]
    [Hermes Gateway] --> [Slack]
    [Hermes Gateway] --> [WhatsApp]
    [Hermes Gateway] --> [Signal]
    [Hermes Gateway] --> [CLI]
    [Telegram] --> [Single Agent Memory]
    [Discord] --> [Single Agent Memory]
    [Slack] --> [Single Agent Memory]
```

**Key benefits:**
- **Team access**: Multiple users interact with same agent
- **Always available**: Agent runs on server, not your laptop
- **Memory continuity**: Same memory across all platforms
- **Voice support**: Automatic voice memo transcription

### 7.2 Platform Setup Comparison

| Platform | Setup Complexity | Voice Support | Group Support | Best For |
|---|---|---|---|---|
| **Telegram** | Low | Full | Yes | Fastest first connection, mobile access |
| **Discord** | Medium | Full | Yes | Team servers, developer communities |
| **Slack** | Medium | Limited | Yes | Workplace integration |
| **WhatsApp** | Medium | Full | DM only | Personal assistant, global reach |
| **Signal** | Low | Full | DM only | Privacy-conscious users |

### 7.3 Telegram Setup Steps

1. Open Telegram, search for `@BotFather`
2. Send `/newbot`, follow prompts to name your bot
3. Copy the bot token (format: `123456789:AAxxxxxxxx`)
4. Configure Hermes:
```bash
hermes gateway add telegram --token "YOUR_BOT_TOKEN"
hermes gateway start telegram
```
5. Send a message to your bot — Hermes responds [^323^] [^326^]

### 7.4 Discord Setup Steps

1. Go to Discord Developer Portal
2. Create "New Application", name it
3. Go to "Bot" section, click "Add Bot", copy token
4. Configure Hermes:
```bash
hermes gateway add discord --token "YOUR_DISCORD_TOKEN"
```
5. Create invite link with Read Messages, Send Messages, Read Message History permissions
6. Invite to server:
```bash
hermes gateway start discord
```

### 7.5 Slack Setup Steps

1. Go to Slack API, create new app
2. Add Bot Token Scopes: `chat:write`, `channels:history`, `groups:history`
3. Install to workspace, copy Bot User OAuth Token
4. Configure Hermes:
```bash
hermes gateway add slack --token "xoxb-YOUR-TOKEN"
hermes gateway start slack
```

### 7.6 24/7 Service Setup

Install gateway as systemd service for persistent operation [^326^]:
```bash
hermes gateway install    # Creates systemd unit
systemctl status hermes-gateway
hermes update             # Keep current — weekly recommended
```

### 7.7 Cross-Platform Continuity

A key feature: start a conversation on Telegram, continue on CLI, finish on Discord — the agent maintains the same session context and memory across all platforms. [^18^] [^323^]

```bash
/model deepseek/deepseek-v4-pro
/model anthropic/claude-opus-4.6
/model openai/gpt-5.5
/model custom:qwen-2.5         # Local model
```

Conversation history, memory, and skills carry over regardless of provider.

### 8.3 Local Model Setup

**Via Ollama** (fully local, zero API cost) [^417^]:
```bash
# Install Ollama
curl -fsSL https://ollama.com/install.sh | sh

# Pull a capable model
ollama pull qwen3.5      # ~11 GB VRAM
ollama pull gemma4       # ~16 GB VRAM

# Set context size
ollama run gemma4 --ctx-size 65536

# Configure Hermes to use Ollama
hermes model
# Select "Custom endpoint"
# API base URL: http://127.0.0.1:11434/v1
# Leave API key blank
```

**Via vLLM/SGLang** (production GPU serving):
```yaml
# ~/.hermes/config.yaml
model:
  default: "NousResearch/Hermes-3-Llama-3.1-70B"
  provider: custom
  base_url: http://localhost:8000/v1
```

### 8.4 Fallback Provider Chains

Configure automatic failover when primary provider fails [^413^] [^417^]:
```yaml
# ~/.hermes/config.yaml
fallback_providers:
  - provider: openrouter
    model: anthropic/claude-sonnet-4
  - provider: ollama
    model: gemma4
```

When activated, fallback swaps model mid-session without losing conversation.

### 8.5 Choosing the Right Setup

| Use Case | Recommendation |
|---|---|
| Just want it to work | OpenRouter or Nous Portal |
| Local, easy setup | Ollama |
| Production GPU serving | vLLM or SGLang |
| Mac / no GPU | Ollama or llama.cpp |
| Cost optimization | OpenRouter with `sort: "price"` |
| Maximum privacy | Ollama, vLLM, llama.cpp (fully local) |
| Multi-provider routing | LiteLLM Proxy or OpenRouter |
| Budget option | MiniMax M2.7 ($0.30/M input), DeepSeek V4 Pro ($0.435/M) |

## 9. Security Model

### 9.1 Zero CVE Record

As of April 2026, Hermes Agent has zero publicly disclosed agent-specific CVEs — compared to OpenClaw's 138 CVEs in 63 days (including 7 critical with CVSS above 9.0). [^34^] [^66^] [^67^]

**Important caveat**: Hermes launched in February 2026, giving it less exposure time. The zero-CVE record is encouraging but not a guarantee. Enterprise security review is still required regardless of CVE history. [^34^]

### 9.2 Defense-in-Depth Architecture

| Layer | Mechanism | Description |
|---|---|---|
| **Prompt injection scanning** | Built-in detection | Scans incoming prompts for injection attempts |
| **Credential filtering** | Context scanning | Strips sensitive env vars from child processes |
| **Container hardening** | Sandbox backends | Read-only root filesystem, dropped capabilities, PID limits |
| **Namespace isolation** | Subagent separation | Each subagent runs in isolated namespace |
| **Command approval** | Pattern matching | Detects destructive commands (recursive deletes, sudo, permission changes) |
| **Pluggable approval gates** | Configurable | Admin can require approval for sensitive operations |
| **Skill manifest signing** | Supply chain | Self-generated skills eliminate third-party marketplace risk |

### 9.3 Independent Security Audit

An independent security audit of v0.8.0 (812 Python files, ~364K lines of code) found [^319^]:
- **No malware or data exfiltration** — code described as "well-intentioned"
- **4 critical and 9 high-severity findings** in default configuration
- Primary issue: default security posture is ALLOW-ALL (permissive by default)

### 9.4 Container Security

Six terminal backends with varying isolation levels [^18^] [^20^]:

| Backend | Isolation Level | Best For |
|---|---|---|
| **Local** | None | Trusted environments only |
| **Docker** | OS-level (sandboxed) | Default recommended |
| **SSH** | Network-isolated | Remote server management |
| **Singularity** | HPC-compatible | High-performance computing |
| **Modal** | Serverless, hibernates | Zero-cost-when-idle |
| **Vercel Sandbox** | Cloud sandbox | Ephemeral execution |

Docker backend security:
```bash
hermes config set terminal.backend docker   # Enable container isolation
# Includes: read-only root, dropped capabilities, no privilege escalation, PID limits
```

### 9.5 Gateway Security

Before running 24/7 with gateway access [^326^]:
1. **Switch to Docker backend**: `hermes config set terminal.backend docker`
2. **Configure allowlist**: Restrict which users can send commands via `~/.hermes/config.yaml`
3. **Lock down BotFather settings**: Disable "Allow Groups" unless explicitly needed

### 9.6 Security Comparison

| Dimension | Hermes Agent | OpenClaw |
|---|---|---|
| **CVEs (as of April 2026)** | 0 | 138 in 63 days |
| **Critical CVEs** | 0 | 7 (CVSS >9.0) |
| **Skill supply chain** | Self-generated (trusted) | Community marketplace (1,467 malicious of 5,700 audited) |
| **Default gateway auth** | Required | Off by default |
| **Sandbox isolation** | 6 backends with namespace isolation | Application-level checks only |
| **Exposed instances** | Not reported | 135,000+ across 82 countries |
| **Default config posture** | ALLOW-ALL (requires hardening) | Weak (described by researchers) |

### 9.7 The Real Attack Surface

Enterprise security analysis identifies these broader risks beyond disclosed CVEs [^34^] [^73^]:
- **Memory injection**: Malicious content in persistent memory channels
- **MCP trust boundaries**: Compromised MCP servers gaining tool access
- **YOLO mode**: Ability to disable all approval prompts creates risk in CI/CD
- **Credential exposure**: API keys stored in `~/.hermes/.env` require file-system protection
- **CVE-2026-7396**: One low-impact path traversal in WeChat adapter (post-April 2026)

## 10. Comparison: Hermes vs Commercial Agents

### 10.1 The AI Agent Stack of 2026

The consensus among analysts: there is no "single AI agent that does it all." Professional teams run **stacks** of multiple agents. [^347^] [^351^] [^352^]

**Recommended stacks:**

| Stack Pattern | Best For |
|---|---|
| **Cursor + Claude Code** | Pure coding workflows |
| **Hermes only** | Non-dev PM/Ops teams using chat |
| **Hermes + Claude Code** | Dev teams in terminal + automation |
| **Cursor + Claude Code + Hermes** | Solo dev / small startup wanting full coverage |

### 10.2 Hermes vs Claude Code

| Dimension | Hermes Agent | Claude Code |
|---|---|---|
| **Type** | General autonomous agent | Terminal coding assistant |
| **Focus** | Persistent automation, learning | Code generation, refactoring, debugging |
| **Memory** | Cross-session persistent (auto) | Session-only (CLAUDE.md for project context) |
| **Learning** | Auto-generates skills | None (stateless) |
| **Multi-platform** | 15+ messaging platforms | Terminal only |
| **Scheduling** | Built-in cron | None |
| **Cost** | Free (MIT) + API costs | $20-200/mo subscription |
| **Best for** | Long-running workflows, automation | Deep codebase understanding, multi-file refactor |

**Verdict**: These are complementary, not competitive. Use Claude Code for coding tasks, Hermes for persistent automation. [^34^] [^347^]

### 10.3 Hermes vs Cursor

| Dimension | Hermes Agent | Cursor |
|---|---|---|
| **Type** | General autonomous agent | AI-native IDE (VS Code fork) |
| **Interface** | CLI + messaging platforms | Full IDE |
| **Autocomplete** | No | Best on market |
| **Code editing** | Via terminal tools | Inline, real-time |
| **Multi-platform** | 15+ messaging | IDE only |
| **Cost** | Free (MIT) + API | $20-200/mo |
| **Best for** | Automation, scheduling | Daily coding, autocomplete |

**Verdict**: Cursor is a coding tool; Hermes is an automation platform. Different use cases entirely. [^34^] [^351^]

### 10.4 Hermes vs OpenClaw

The most direct comparison. Both are open-source, self-hosted, general-purpose agents. [^34^] [^35^] [^64^] [^319^]

| Dimension | Hermes Agent | OpenClaw |
|---|---|---|
| **Philosophy** | Agent-first (runtime-centric) | Gateway-first (communication-centric) |
| **Learning** | Self-improving closed loop | Static human-written skills |
| **Memory** | Three-layer persistent, auto-managed | Basic text files, manual config |
| **Security (CVEs)** | 0 | 138 |
| **Skill origin** | Auto-generated by agent | Human-written + marketplace |
| **Setup time** | 2-4 hours | <30 minutes |
| **Stars** | 100K+ | 346K+ |
| **Best for** | Deep work, learning, automation | Quick deployment, broad messaging |

**They can work together**: MCP and A2A protocol support means OpenClaw can handle channel routing while Hermes handles intelligent task execution. [^319^]

### 10.5 Lock-in Comparison

| Tool | Vendor Lock-in | How Hard to Switch |
|---|---|---|
| **Hermes Agent** | Lowest (MIT + agentskills.io standard) | Easiest — open skill format |
| **OpenClaw** | Low (MIT) | Easy — BYOM |
| **Claude Code** | Medium (tied to Anthropic) | Medium — CLAUDE.md becoming standard |
| **Cursor** | High (proprietary IDE) | Medium — can return to VS Code |

## 11. Curriculum Exercises & Projects

### 11.1 Beginner Exercises (Week 1)

**Exercise 1.1: Install and Verify** (~30 min)
1. Run the one-line Hermes Agent installer
2. Verify with `hermes version`, `hermes doctor`, `hermes status`
3. Document any errors and troubleshooting steps taken

**Exercise 1.2: First Conversation** (~15 min)
1. Start Hermes with `hermes`
2. Ask: "What tools do you have available?"
3. Try: "Read my home directory and list the 5 most recently modified files"
4. Observe the tool call pattern in the output

**Exercise 1.3: Model Exploration** (~30 min)
1. Set up OpenRouter provider (get free API key)
2. Try three different models via `/model` command
3. Compare response quality and speed across models
4. Document findings

**Exercise 1.4: Skill Observation** (~30 min)
1. Give Hermes a multi-step task: "Check GitHub trending repos and summarize"
2. After completion, examine `~/.hermes/skills/`
3. Read the auto-generated skill file
4. Identify what patterns the agent extracted

### 11.2 Intermediate Exercises (Week 2)

**Exercise 2.1: Telegram Bot Setup** (~45 min)
1. Create bot via @BotFather
2. Configure gateway: `hermes gateway add telegram --token "..."`
3. Send messages from phone, observe responses
4. Test memory: mention a preference, start new conversation, verify recall

**Exercise 2.2: Cron Automation** (~45 min)
1. Create a daily report task:
```
hermes cron add "every day at 9am" --task "Summarize top 3 AI news from Hacker News"
```
2. Test with manual trigger
3. Verify delivery to Telegram
4. List and monitor with `hermes cron history`

**Exercise 2.3: Local Model Setup** (~60 min)
1. Install Ollama
2. Pull Qwen 2.5 Coder 32B
3. Configure Hermes to use local endpoint
4. Compare latency and quality vs. cloud models
5. Calculate cost savings

**Exercise 2.4: MCP Integration** (~45 min)
1. Add GitHub MCP server to `~/.hermes/config.yaml`
2. Test: "List my recent GitHub issues"
3. Add filesystem MCP server
4. Test cross-tool integration

### 11.3 Advanced Projects (Week 3-4)

**Project 3.1: Multi-Platform Personal Assistant**
- Set up Hermes on Telegram, Discord, and CLI simultaneously
- Configure persistent memory for user preferences
- Create cron jobs for daily briefings delivered to preferred platform
- Document cross-platform continuity experience
- **Deliverable**: Working multi-platform agent with 3+ scheduled automations

**Project 3.2: Self-Improving Research Agent**
- Configure Hermes for automated research on a topic of choice
- Use subagent delegation for parallel research streams
- Observe skill generation over 20+ tasks
- Analyze how skills improve the reflection cycle
- **Deliverable**: Report on skill evolution with before/after comparison

**Project 3.3: Security Hardening Audit**
- Review default Hermes configuration for security issues
- Implement Docker sandboxing for terminal commands
- Configure gateway allowlists
- Perform threat modeling for your specific deployment
- **Deliverable**: Security hardening checklist with implemented mitigations

**Project 3.4: Comparative Framework Analysis**
- Install and configure Hermes Agent, OpenClaw, and Aider side by side
- Execute identical tasks on each framework
- Measure: setup time, task completion quality, cost, security posture
- **Deliverable**: Comparative analysis report with recommendation matrix

1. **Architecture**: Why did Nous Research choose SQLite + FTS5 over vector databases for memory? What are the trade-offs?
