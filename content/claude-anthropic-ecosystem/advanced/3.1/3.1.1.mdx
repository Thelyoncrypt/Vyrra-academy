## 5.1 Setting Up Claude Code

```bash
# Install globally
npm install -g @anthropic-ai/claude-code

# Run in a Git repository
cd my-project
claude

# First run will authenticate via browser
```

## 5.2 Plan Mode vs. Act Mode

**Key Concepts:**

### Plan Mode (`/plan`)

- Claude analyzes the task and creates a step-by-step plan

- You review and approve before any code changes

- Best for: Complex refactors, new features, architectural changes

- Prevents mistakes by adding a review checkpoint

### Act Mode (Default)

- Claude directly implements changes

- Faster for simple, well-defined tasks

- Best for: Bug fixes, small additions, documentation

### Toggling Between Modes

```bash
/plan     # Switch to plan mode
/act      # Switch to act mode
# Or use Shift+Tab to toggle
```

### Best Practice

> Always start complex work in Plan Mode. Review the plan carefully. Only switch to Act Mode after you understand and agree with the approach.

## 5.3 Context Management Strategies

**Key Concepts:**

- Context window: The amount of text Claude can "see" at once

- Token budget: Every file read, every message consumes tokens ($)

- Efficient context = lower costs + better results

- CLAUDE.md: Project-level instructions and context

- `/clear`: Reset context when it gets bloated

- `/context`: View current token usage

### CLAUDE.md Best Practices

```markdown
# CLAUDE.md — Project Configuration

## Project Overview
Brief description of what this project does.

## Tech Stack
- Language: TypeScript
- Framework: Next.js 14 (App Router)
- Database: PostgreSQL with Prisma ORM
- Styling: Tailwind CSS
- Testing: Vitest + Playwright

## Architecture Rules
- All API routes go in app/api/
- Database queries only in lib/db/
- Components in components/ (server) or components/client/
- Never use 'any' type — always define interfaces

## Code Style
- Use named exports, not default exports
- Prefer async/await over .then()
- Error handling: Always wrap external calls in try/catch
- Use the existing logger in lib/logger.ts

## Commands
- `npm run dev` — Start development server
- `npm test` — Run unit tests
- `npm run e2e` — Run Playwright tests
- `npm run lint` — Run ESLint
- `npm run db:generate` — Generate Prisma client
- `npm run db:migrate` — Run database migrations
```

## 5.4 Custom Commands & Automation

**Key Concepts:**

- Slash commands (`/command`) for quick actions

- Custom commands defined in `.claude/commands.json`

- Commands can run shell scripts, open files, or trigger workflows

- Combine with Git hooks for automation

### Creating a Custom Command

```json
// .claude/commands.json
{
  "commands": [
    {
      "name": "test-changed",
      "description": "Run tests for changed files only",
      "command": "npm test -- --changedSince=HEAD~1"
    },
    {
      "name": "lint-fix",
      "description": "Run linter with auto-fix",
      "command": "npm run lint -- --fix"
    }
  ]
}
```

## 5.5 MCP Server Integration

**Key Concepts:**

- MCP (Model Context Protocol) extends Claude Code's capabilities

- 10,000+ community-built MCP servers

- Categories: Databases, APIs, file systems, web scraping, etc.

- Install via `/mcp add <server>` or manual configuration

- Popular MCP servers: PostgreSQL, Supabase, Brave Search, GitHub

### Installing MCP Servers

```bash
# List available MCP servers
/mcp

# Add a specific server
/mcp add mcp-server-postgres

# Configure in .claude/mcp.json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://localhost/mydb"]
    }
  }
}
```

## 5.6 GitHub Workflows

**Key Concepts:**

- Claude Code integrates with GitHub for PR creation and review

- GitHub Actions can trigger Claude for automated code review

- `/github` command for PR operations

- Best practice: Always review Claude's changes before committing

### Claude + GitHub Actions Integration

```yaml
# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude Code Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          npm install -g @anthropic-ai/claude-code
          claude review-pr --pr ${{ github.event.pull_request.number }}
```

## 5.10 Subagents & Advanced Patterns

**Key Concepts:**

- **Subagents**: Separate Claude Code instances for isolated tasks

- **Parallel execution**: Run multiple Claude instances simultaneously

- **Git worktrees**: Isolate different feature branches

- **Hooks**: Intercept and modify Claude's behavior

- **PostToolUse hooks**: Auto-format code after file edits

### Running Multiple Claude Instances

```bash
# Terminal 1: Working on feature A
claude

# Terminal 2: Working on feature B (different worktree)
git worktree add ../feature-b feature-branch
claude

# Terminal 3: Running tests
claude /test

# Use iTerm split panes for visibility
```

## 5.11 Best Practices & Common Pitfalls

### Best Practices

1. **Always use Git**: Claude Code requires Git - commit frequently

2. **Keep CLAUDE.md updated**: It's a living document

3. **Use Plan Mode for complex tasks**: Prevents costly mistakes

4. **Monitor token usage**: `/context` shows current spend

5. **Clear context regularly**: `/clear` when switching tasks

6. **Verify all outputs**: Review every change before committing

7. **Use verification commands**: Tell Claude how to verify its work

8. **Start features in Plan Mode**: Always plan before acting

### Common Pitfalls

1. **Context bloat**: Too many files open → poor performance, high cost

2. **Trusting hallucinations**: Always verify code changes

3. **No verification**: Claude can't always test its own code

4. **Outdated CLAUDE.md**: Stale instructions cause confusion

5. **Too many skills**: Focus on 5-10 high-quality skills

6. **Ignoring thinking blocks**: Read Claude's reasoning for better results
